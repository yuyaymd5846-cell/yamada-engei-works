# アプリ全体レビュー（2026-04-03）

対象: `yamada-engei-works`（Next.js 15 + Prisma + Supabase）

## 1) まず直すべき点（品質・不具合リスク）

### 1-1. ESLint エラーが大量に残っている
- `npm run lint` が通らない状態です。
- 主に `any` の多用、未使用変数、Hook 依存配列の不足、`<img>` 利用が要因です。

**優先度: 高**（CI の健全化・保守性）

---

### 1-2. `DELETE /api/record?all=true` が無条件で全削除できる
- API 側に認可チェックがなく、URL パラメータのみで全レコード削除可能です。

**優先度: 最高**（運用事故・セキュリティ）

**対策案**
- 管理者ロール判定（Supabase Auth / JWT）を必須化。
- `all=true` は開発環境のみ許可、あるいは機能自体を削除。
- 操作履歴（監査ログ）を残す。

---

### 1-3. 主要 API が「全件取得→メモリ内集計」になっている
- `/api/analysis` は `cropCycle` 全件 + `workRecord` 全件を取得し、Node.js 側でフィルタ・集計しています。
- データ増加時に応答遅延・メモリ増加が顕著になります。

**優先度: 高**（性能・スケーラビリティ）

**対策案**
- 期間・ハウス・作目を SQL where 条件へ移す。
- `groupBy` / 集計クエリで DB 側集約。
- 必要なカラムだけ `select`。

---

### 1-4. レコード一覧画面が全件をクライアントで保持・加工
- `/records` は `/api/record` から全件取得し、月フィルタ・ソート・ページングをブラウザ側で実施。
- 件数が増えると初回表示・操作レスポンスが悪化します。

**優先度: 高**（体感速度）

**対策案**
- サーバー側ページング（`limit/offset` または cursor）へ移行。
- フィルタ条件（年月・ハウス・作業）を API クエリに渡して DB で絞る。
- 可能なら `react-virtualized` 系でテーブル仮想化。

---

### 1-5. N+1 / 重複計算が多いダッシュボードロジック
- ダッシュボードで複数テーブルを広く読み、アプリ側でループ・マッピングしています。
- 営農データ増加時に SSR の遅さにつながります。

**優先度: 中〜高**

**対策案**
- 日次バッチで「今日の推奨作業」を事前計算しキャッシュ。
- 取得列を最小化。
- 日付計算・判定ロジックをユーティリティ化してテスト可能に。

## 2) 軽くするための実装優先順位（現実的な順）

1. **分析 API の DB 集計化**（全件ロードをやめる）
2. **記録一覧 API のサーバー側ページング**
3. **画像表示を `next/image` へ統一**（LCP 改善）
4. **重い集計結果に短TTLキャッシュ**（`revalidate`, `unstable_cache`）
5. **Prisma のインデックス見直し**（`date`, `greenhouseName`, `batchNumber`, `workName`）

## 3) 具体的な改善イメージ

### 3-1. `/api/record` の GET にクエリ対応を追加
- 例: `?month=2026-04&gh=①-S&work=定植&page=1&pageSize=50`
- DB where + orderBy + pagination で返却。
- フロント側は `fetchRecords` でクエリ付き再取得。

### 3-2. `/api/analysis` を SQL 集計中心に再設計
- まず対象レコードだけを where で絞る。
- 日次トレンドは日付 + 作業名単位で集計。
- ハウス比較は greenhouse 単位で集計。
- `allCycles.find(...)` の線形探索をなくす。

### 3-3. 一覧画面の画像を最適化
- `<img>` を `next/image` に置換。
- サムネイルとオリジナルを分離。
- Supabase Storage の transform で軽量サイズを配信。

## 4) 運用安定性の観点で追加したいこと

- API の入力バリデーション（`zod`）
- 監査ログ（削除系 API）
- `npm run lint` / `npm run build` を CI 必須化
- Prisma クエリの実行時間ログ（遅いクエリを可視化）

## 5) すぐ着手するならこの3つ

1. `/api/record` 全削除APIのガード追加
2. `/api/analysis` の全件取得をやめる
3. `/records` のサーバー側ページング化

---
